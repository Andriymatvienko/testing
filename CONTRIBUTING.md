# Інструкція для розробників та контриб'юторів (Contributing Guidelines)

Перш за все, дякуємо вам за інтерес до покращення нашого консольного нотатника! Спільнота розробників робить цей проєкт кращим.

Цей документ містить звід правил, рекомендацій та стандартів, які допоможуть вам швидко розгорнути середовище розробки та зробити свій внесок у проєкт максимально ефективним.

---

## З чого почати?

### 1. Повідомлення про баги (Bug Reports)
Якщо ви знайшли помилку (функціональну або UI/UX дефект в консолі):
* Переконайтеся, що цього багу немає у списку відкритих Issue.
* Створіть новий тикет за допомогою стандартного шаблону **Bug Report**.
* Обов'язково додайте кроки для відтворення (Steps to Reproduce), очікувану та фактичну поведінку застосунку, а також лог помилки (Traceback) або скриншот консолі.

### 2. Запити на нову функціональність (Feature Requests)
Якщо у вас є ідея, як покращити нотатник (наприклад, додати шифрування нотаток, категорії або теги):
* Створіть тикет типу **Feature Request**.
* Детально опишіть проблему, яку вирішує ваша пропозиція, та бачення її реалізації в коді.

---

## Процес розробки (Workflow)

Для того щоб внести зміни у вихідний код програми, дотримуйтеся наступного Git-процесу:

1. **Зробіть Fork проєкту** у свій особистий профіль на GitHub.
2. **Створіть нову гілку** від актуального стану гілки `main`. Назва гілки має відображати суть змін:
   ```bash
   git checkout -b feature/назва-нової-фічі
   # або для багфіксів:
   git checkout -b bugfix/опис-помилки
   ```
3. Внесіть зміни у код. Намагайтеся дотримуватися стилю кодування PEP 8 (іменування змінних, відступи, наявність коментарів).
4. Напишіть або оновіть тести. Якщо ви додаєте новий функціонал, обов'язково додайте відповідні тест-кейси у файл test_notebook_pytest.py.
## Правила тестування перед відправкою
Кожен розробник зобов'язаний перевірити свій код локально перед відправкою на GitHub. Усі наявні тести повинні мати статус PASSED.

Запуск тестування:
```bash
pytest test_notebook_pytest.py -v
```
Ваш Pull Request не буде затверджено, якщо хоча б один тест повертає помилку (FAILED), яка не була задекорована як очікувана (@pytest.mark.xfail).

### Стандарти для коммітів (Commit Messages)
Повідомлення до ваших коммітів мають бути чіткими, зрозумілими та написаними в теперішньому часі.
Рекомендована структура: Тип: Короткий опис змін
Приклади правильних коммітів:
```bash
feat: add encryption for note text fields — додавання нової фічі.

fix: resolve crash in delete_note when input is alphabetical — виправлення багу.

docs: update setup steps in README — оновлення документації.

test: incorporate unit tests for sorting capability — додавання тестів.
```
## Створення Pull Request (PR)
Після того як ви зафіксували всі зміни локально та відправили їх у свій форк (git push origin feature/назва-фічі):
1. Перейдіть на сторінку оригінального репозиторію на GitHub.
2. Натисніть кнопку Compare & pull request.
3. Опишіть у коментарях до PR, які саме завдання чи Issue вирішує ваш код.
4. Очікуйте на рецензію (Code Review) від ментора або власника репозиторію. Якщо будуть зауваження — внесіть виправлення у ту саму гілку.
